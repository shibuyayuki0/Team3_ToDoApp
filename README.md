<br/>
本システムは、チーム進捗などのタスクを名称・優先度・期日で管理するためのWEBアプリケーションです。  
タスクを一覧で表示し、タスクの状態を未完了・完了別に管理することができます。  
基本機能を満たしつつ、自分たちの考えるToDoアプリの開発を行いました。

# 機能
## 基本機能
- タスクを一覧で表示する
- タスクの状態（「すべて」・「未完了」・「完了」）でフィルター表示
- 1件のタスクに対し、タスク名・期限・詳細内容・優先度を登録
- 既存のタスクを編集・更新・削除が可能
- 一覧画面上でワンクリックでタスクの完了⇔未完了の切り替えが可能

## 追加の機能
- タスクの優先度別に色分けをして表示
- タスクの期日を昇順で表示する（期日が近いものが一番上に表示）
- タスクにディレイがあると、期日の横にアラート表示

## タスクの登録
1. 【タスク一覧】画面から「タスク作成ボタン」をクリックします。
2. 【タスクの新規作成】画面で登録したいタスクを入力します。
    - タスク名　：　30文字以下でタスクの名称を記載します。<font color="Red">*必須</font>
    - 優先度　　：　高・中・低でタスクの優先度を設定します。
    - 期限　　　：　タスクの期限を入力します。入力欄右側のカレンダーアイコンで、GUI入力も可能です。*必須
    - 詳細内容　：　タスクの詳細事項を記入します。
3. 【タスクの新規作成】画面内の「タスク作成ボタン」をクリックします。

## タスク表示の切り替え（フィルター）
1. 【タスク一覧】画面でフィルターをクリックし、プルダウンメニューを開きます。
2. 表示したいタスクの一覧を選択します。

## タスク状態の切り替え
1. 【タスク一覧】画面で任意のタスクの、タスク完了ボタンをクリックします。

## タスクの編集
1. 【タスク一覧】画面で任意のタスクをクリックします。
2. 【タスクの詳細】画面で、「タスク編集ボタン」をクリックします。
3. 【タスクの編集】画面で、タスクの内容を修正します。
4. 【タスクの編集】画面の「タスク更新ボタン」をクリックします。

## タスクの削除
1. 【タスク一覧】画面で任意のタスクをクリックします。
2. 【タスクの詳細】画面で、「タスク削除ボタン」をクリックします。

# 表示・機能上のアピールポイント
## タスク一覧をカード状にシンプル表示
【タスク一覧】画面は、完了ボタン・タスク名・期限だけの表示として、シンプルに構成しました。  
優先度は、高（赤）→中（黄）→低（緑）で色分け表示し、直感的に分かりやすいようにしました。  
また、完了タスクについても、タスク色の変更や打消し線などで、完了済みであることを分かりやすく表示しています。

## 詳細画面
【タスク一覧】画面でタスクをクリックすることで、詳細画面が表示されます。  
一覧には表示されていない、タスクの詳細内容などはこちらから確認でき、そのまま編集画面に遷移することが可能です。  
<br/>
【タスクの詳細】画面の表示中は、画面背景（タスク一覧）を暗くすることで表示を見やすくしています。  
また、画面背景部をクリックすることで、「閉じるボタン」を押さずとも【タスクの詳細】画面を非表示にすることが可能です。

## アラート表示
未完了のタスクの期限が、過去になっている場合は、期限の横にアラートマークを表示します。

## スクロールボタンの実装
タスクの表示が画面を見切れるとき、一定以上の画面スクロールを行うとTOPに戻るボタンが表示されます。

## レスポンシブ対応
スマホやPCなど、画面サイズが変わっても、表示が大きく崩れないようにしました。  
例えば、画面サイズが一定以下になるとフィルターが2行目に表示されます。

## テスト定義と実施

## 【タスクの新規作成】画面と【タスクの編集】画面の共通化
両者の画面は、共通のソース（ToDoEditor）をベースに、起動時のタスクIDの有無で、一部文言や処理内容を切り替えて動作します。
これにより、画面の統一感が自動的に担保され、また修正時のコストが最小限になります。  
編集と新規作成の切り替えには、[ページディレクティブ](#ページディレクティブ)を活用しました。

## フィルタ切替後のリダイレクト
フィルターは、条件を変えてのSQLからデータ再取得を実行するため、POSTで操作をします。  
このとき、再表示をOnPost内で行わず、リダイレクトをすることで、OnGetが走り再表示が実行されます。
また、ユーザーがブラウザをリロードしても、OnGetが再実行されるだけなので、再送信ダイアログなども表示されません。

## インターフェースの活用
SQL操作部分をインターフェース化することにより、画面実装者とDB操作部実装者の間で、お互いの詳細な仕様をブラックボックスにしたまま、データ送受信が可能です。  
これにより、画面実装者はDBからデータを受け取ったあと、あるいはデータを書き込む直前までを注力して開発でき、DB操作部実装者は自身がどう使用されるのかは知らずに、DBとの必要な情報のやり取りにのみ注力して開発が可能です。

## ページディレクティブの活用
編集画面と新規作成画面の切り替えには、画面呼び出し時にタスクIDを付与することで実現しました。  
<br/>
今回はIDとはデータにアクセスするキーなので、クエリパラメータではなく、パスに埋め込む形で取得を試みます。
まず、以下のようにToDoEditor.cshtmlにURLルーティングを記載します。  
今回であれば、"/edit"でアクセスし、その後ろに"/{id}"を省略可能でつけます。

``` html
### ToDoEditor.cshtml
@page "/edit/{id?}"
@model ToDoEditorModel

```

<br/>
つぎに、C#側でOnGetメソッドを実装します。  
OnGetメソッドでは、同じ名称のNULL許容string型引数を受けるようにします。  
その後、メソッド内で分岐することで、IDありとIDなしの処理をまとめることができます。
<br/>

``` csharp
### ToDoEditor.cshtml.cs
...（前略）...

public IActionResult OnGet(string? id) {

...（以下省略）...

if (!string.IsNullOrEmpty(id))
{
  /* IDがInt変換可能かつ範囲内インデックスであるか？ */
  /* IDが空じゃないときの処理＝編集・更新 */
}
else
{
  /* IDが空のときの処理＝新規作成 */
}
```

<br/>
そして、実際にページをアクセスすると、以下のようにあたかもそういったページにアクセスしているかのようにふるまうことができます。
<br/>
<img width="368" height="88" alt="image" src="https://github.com/user-attachments/assets/6500da7d-7b88-4b0a-8f5e-3b0d1cdfefa6" /><br/>
<br/>
また、このときのidはURLとして残っているので、Postリクエスト（更新ボタンなど）を発行したときもこのIDは引き継がれるので、
<br/>

``` csharp
### ToDoEditor.cshtml.cs
...（前略）...

public IActionResult OnPost(int? id) {
  /* Post時の処理 */
}
```

<br/>
のように、IDを受け取り直すことも可能です。  
今回であれば、IDなし＝ADD（新規作成）、IDあり＝UPDATE（更新）のように分岐します。  
なお、OnGetですでにIdは、正常な範囲のINT型である（不正な文字列や範囲外インデックスを排除）と確定させているので、OnPost側ではInt?として受け取っています。

## JavaScriptとの連携
画面遷移が一覧⇔編集・新規作成しかないのに対し、【タスクの詳細】画面の表示にはJSを活用しました。  
また、詳細表示は、クリックしたタスクからタスクIDを取得し、これをJS側で処理します。  
  - 編集ボタンのhrefに、編集対象となるタスクIDを付与する
  - 削除ボタン内に、削除対象タスクIDを埋め込む
  - 各タスク項目を取得して、画面に表示する

## 学んだこと
- Razor Pagesを使うことで、C#とHtml、JavaScriptとの連携を一通り体験した。
  - WEBアプリケーションを作るにあたって、さまざまな技術が必要で、それぞれに特化した人材が必要であると感じた。
- SQLをC#から発行し、DBをプログラムとして操作することを体験した。
  - プログラムで正確にデータを取得しようとすると、SQL文を正しく発行する必要があり、これを怠ると思った結果にならなかった。
  　　- 例:優先度リストをDBからSELECTした際、ORDER BYを掛けなかったために思った通りの順序（高→低）にならなかった
- チームでの作業において、GitHubを活用することで開発データの整合性や作業競合の解消などを一通り体験できた。
  - ケース1：全体に影響するバグを修正・マージしたあと、各人のブランチにローカルmainを逆にマージして最新化 → Pushしてリモート側にも反映した
  - ケース2：お互いにIndex.cshtmlを編集することで不整合・競合が発生したため、会話をしながら解消をした
  - ケース3：ブランチのブランチをマージした後、マージ後のブランチでコミットをかけて変更を上書きしてしまったので、そのコミットを無かったことにした（差し戻した）
- モンキーテスト、中間報告などで通常ではありえない動作かつ想定外の挙動が起きたため、これを直す経験をした
  - 例：タスクをクリックをした回数に応じて、遷移先のURLが変化してしまう
    - これはクリックするとURLにIDを付与する制御にしていたので、クリックしても画面遷移をしないとこれが残っていたため
    - 自身の実装方法に由来して発生しており、見直しや単体でのチェックが必要だと感じた

- 当初予定していた、残り時間や遅れ時間を表示する機能が未実装である。（スケジュールの都合）

  - たとえば、並び替えルールの切り替え機能があってもいいと思う
